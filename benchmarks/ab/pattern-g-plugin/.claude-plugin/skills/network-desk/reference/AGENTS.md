# AGENTS.md — Network Desk Knowledge Vault

This is the **schema and operating manual** for the **network-desk knowledge vault** — the curated corpus that backs the cloud-networking specialists. Every agent (and every human contributor) must read this before writing into the vault. Pattern reference: <https://gist.github.com/karpathy/442a6bf555914893e9891c11519de94f>.

## Purpose

The vault is the **knowledge layer** of the network-desk extension. It contains opinionated, distilled, cross-referenced notes on cloud networking — Azure, AWS, GCP, and the 14 firewall vendors network-desk supports. It is consumed at runtime by the `cn_search` tool (BM25 + 1-hop wikilink expansion) and at author-time by humans browsing it in Obsidian (the directory layout, frontmatter, and `[[wikilinks]]` are Obsidian-compatible by design).

The vault answers:

- "What is service X, when do I use it, when do I avoid it?"
- "How does topic Y (BGP, hybrid DNS, IP planning, TLS offload, …) work across clouds?"
- "What is the reference architecture for pattern Z?"
- "What's vendor V's syntax / capability / gotcha for capability C?"

It is **not** a docs mirror. Microsoft Learn, AWS docs, GCP docs already exist and are linked, not copied. The vault is the **synthesis** layer.

## 3-Layer Model

1. **Raw sources** (immutable, outside the vault) — vendor docs, RFCs, vendor PG announcements, lab repos, blog posts. Linked, never copied.
2. **Vault** (this directory) — synthesis pages with YAML frontmatter, heavy use of `[[wikilinks]]`, source pointers back to (1).
3. **Personas** (`extensions/network-desk/specialists/<name>/agents/<name>.md`) — the 20 specialist role definitions. They reference the vault by tag and path; they are *not* part of the vault itself.

## Vault Structure

```
vault/
├── AGENTS.md                  # this file
├── _Index.md                  # top-level entry page
├── _migrated/                 # (transient) staging populated by tools/migrate-skills-to-vault.mjs
│   └── <specialist-dir>/      # one folder per original specialist
│       └── <skill-name>.md    # one file per original skill, with provenance frontmatter
├── Services/                  # one page per cloud service
│   ├── Azure/
│   ├── AWS/
│   └── GCP/
├── Topics/                    # cross-cutting themes — grouped into domain subfolders
│   ├── CDN/                   #   one subfolder per topic domain
│   ├── DNS/                   #   (Automation, Capacity, CDN, Containers, Diagrams,
│   ├── Firewall/              #    DNS, Firewall, Hybrid, IaC, IPv6, Load-Balancing,
│   ├── …                      #    Monitoring, Multi-Cloud, Pricing, Private-Link,
│   └── Troubleshooting/       #    Reporting, SASE, Security, Troubleshooting, VNet, VWAN)
│       └── _Index.md          #   domain folders may contain an _Index.md decision tree
├── Patterns/                  # reusable reference architectures (flat — small set)
├── Vendors/                   # firewall vendor axis — 14 pages (one per supported vendor),
│                              #   generated by tools/extract-firewall-vendors.mjs
└── Labs/                      # optional — worked examples, case studies
```

`_migrated/` is the **staging area** populated by `tools/migrate-skills-to-vault.mjs`. The semantic refactor (Phase 2) moves pages out of `_migrated/` into `Services/`, `Topics/<Domain>/`, `Patterns/`, or `Vendors/` with proper frontmatter and `[[wikilinks]]`, then `_migrated/` is deleted.

### Topics subfolder convention

- Every page under `Topics/` lives in a domain subfolder. The folder name is short, capitalised, and uses kebab-case for multi-word names (`Load-Balancing/`, `Multi-Cloud/`, `Private-Link/`).
- **Domain-specific `*-Troubleshooting.md` pages live in their domain folder** (e.g. `Topics/CDN/CDN-Troubleshooting.md`, not `Topics/Troubleshooting/CDN-Troubleshooting.md`). The `Topics/Troubleshooting/` folder is reserved for **vendor-agnostic, cross-cutting techniques** (packet capture, MTU/PMTUD, NAT debug, routing debug, TLS handshake debug, latency analysis, connectivity test, PCAP analysis).
- Folder indexes are optional and use the filename `_Index.md` (e.g. `Topics/Troubleshooting/_Index.md` is a thin decision-tree pointing into the right runbook, not an encyclopedia).
- `Services/` and `Patterns/` stay flat — too few pages to justify subfolders.

### Vendors axis convention

The `Vendors/` folder is the **firewall-vendor axis**, complementing the **firewall-topic axis** in `Topics/Firewall/`. The two axes intersect via wikilinks, not by duplication:

- **`Topics/Firewall/<Topic>.md`** holds the cross-vendor concept (e.g. zone-design principles, HA architecture comparison, rule-audit categories). Vendor specifics inside the Topic are collapsed to a bullet list pointing at the Vendor pages: `- **[[Vendors/PAN-OS#Hardening|PAN-OS]]**`.
- **`Vendors/<Vendor>.md`** holds the vendor-specific syntax, CLI commands, and config snippets organised under a uniform H2 outline (`Overview` / `Config generation` / `HA` / `Policy design` / `Hardening` / `Logging` / `Rule audit` / `Policy testing` / `Troubleshooting` / `Common gotchas` / `See also`). Sections with no extracted content are omitted.

A single-vendor customer reads one Vendor page front-to-back to see everything that vendor needs. A cross-vendor decision (e.g. picking between Azure Firewall, AWS Network Firewall, and PAN-OS for a multi-cloud deployment) reads the Topic page and follows wikilinks. **No duplication** between the two axes.

Generated by `tools/extract-firewall-vendors.mjs`; see the template in `Page Templates → Vendors/<Vendor>.md` below for the exact frontmatter and H2 contract.

## Frontmatter Contract

YAML frontmatter is **mandatory** on every page. It drives BM25 field weighting at retrieval time, Obsidian Dataview queries at browse time, and tag-based persona scoping.

### Required fields (all pages)

```yaml
---
type: service          # service | topic | pattern | vendor | lab | skill (skill = migrated, pre-classification)
name: Azure Firewall   # display name
tags: [firewall, security, l7, tls-inspection]
status: ga             # ga | preview | deprecated | needs-classification
updated: 2026-06-01    # ISO 8601
---
```

### Type-specific fields

- **`type: service`** — adds `cloud: azure|aws|gcp|multi` and `category: networking|security|delivery|monitoring`.
- **`type: vendor`** — adds `vendor_kind: cloud-fw|ngfw|sse|oss-fw|host-fw` (deployment / license model) and `roles: [firewall|router|sd-wan|sse, ...]` (functional categorization — see [Roles axis (Vendors only)](#roles-axis-vendors-only) below).
- **`type: pattern`** — adds `clouds: [azure, aws]` (which clouds this pattern applies to).
- **`type: lab`** — adds `date: YYYY-MM-DD`, `repo: <url>`, and `outcome: <one-line>`.

### Persona scoping field (Phase 2)

Every Service / Topic / Pattern / Vendor page also declares which specialist personas claim it. This is the **single source of truth** for `cn_search`'s specialist-scoping filter — personas do **not** maintain their own path lists.

```yaml
specialists: [cn_fw, cn_nsec]    # this page is in-scope for the firewall and network-security personas
```

### Provenance fields (only on `_migrated/` pages)

```yaml
migration_source: extensions/network-desk/specialists/firewall-engineer/skills/rule-audit/SKILL.md
original_specialist: firewall-engineer
original_skill: rule-audit
```

These are deleted when the page is reclassified out of `_migrated/`.

### `Vendors/<Vendor>.md`

Pages in `Vendors/` are the **firewall vendor axis** of the vault. They hold vendor-specific syntax, CLI commands, and config snippets that would otherwise duplicate across every `Topics/Firewall/*.md` page. Cross-vendor concepts (zone design, HA architecture, audit categories) stay in `Topics/Firewall/`; the Topic pages link to the Vendor pages via wikilink bullets.

Generated and maintained by `tools/extract-firewall-vendors.mjs`. The H2 outline below is the canonical shape every Vendor page MUST follow — the extractor enforces it. Sections with no extracted content are omitted (don't emit empty headings).

```yaml
---
type: vendor
name: PAN-OS (Palo Alto Networks)
vendor_kind: ngfw                  # cloud-fw | ngfw | sse | oss-fw | host-fw
roles: [firewall, sd-wan]          # firewall | router | sd-wan | sse — alphabetical, no implicit primary
tags: [firewall, vendor, palo-alto, pan-os, ngfw]
specialists: [cn_fw]
status: stable
updated: 2026-06-01

# PAN-OS (Palo Alto Networks)

## Config generation       # extracted from Topics/Firewall/Firewall-Config-Generation.md
## HA                      # extracted from Topics/Firewall/Firewall-HA-Design.md
## Policy design           # extracted from Topics/Firewall/Firewall-Policy-Design.md
## Hardening               # extracted from Topics/Firewall/Firewall-Hardening.md
## Logging                 # extracted from Topics/Firewall/Firewall-Log-Analysis.md
## Rule audit              # extracted from Topics/Firewall/Firewall-Rule-Audit.md
## Policy testing          # extracted from Topics/Firewall/Firewall-Policy-Testing.md

```

To regenerate Vendor pages after editing a Firewall Topic: re-run `node tools/extract-firewall-vendors.mjs`. The extractor is idempotent — re-running on already-collapsed topics finds 0 H3s and skips the rewrite step. Hand-authored Overview and Common gotchas sections **are preserved** across re-runs via the `<!-- AUTO-STUB -->` marker: as long as the marker is gone (i.e. you authored real content), the extractor reads the existing section and re-emits it verbatim. Other H2 sections (Config generation, HA, Policy design, …) are always regenerated from the source Topics — don't hand-edit those; edit the source Topic instead.

### Roles axis (Vendors only)

Modern network appliances frequently span multiple product categories — FortiGate is both a firewall and an SD-WAN platform; Juniper SRX is a router with security services; OPNsense / pfSense / VyOS are edge routers as much as they are firewalls. The `roles:` frontmatter field captures this on every Vendor page so cross-axis retrieval (e.g. "list everything that can act as an SD-WAN device") is possible without changing folder structure.

**Controlled vocabulary** (extend cautiously — every new value forces a manual decision on every page):

| Role | Meaning |
|---|---|
| `firewall` | Stateful packet inspection / NGFW as a primary deployment justification |
| `router` | Dynamic routing (BGP/OSPF/MPLS/IS-IS) as a primary deployment justification |
| `sd-wan` | Overlay-tunnel orchestration with central path selection |
| `sse` | Security Service Edge — cloud-delivered SWG / ZTNA / CASB |

**Semantics:** `roles:` lists *every* deployment-justifying role the platform fulfills, regardless of which folder the page lives under. Order is alphabetical (no implicit primary). VPN, IPS, NAT, decryption, etc. are *features* present in most modern firewalls — they are not enumerated here.

**Source of truth:** `VENDORS_META[<vendor>].roles` in `tools/extract-firewall-vendors.mjs`. The extractor emits this field on every regeneration. To change a vendor's roles permanently, edit the extractor and re-run; ad-hoc hand edits to the frontmatter survive until the next recovery cycle and then get overwritten by the extractor's value.

**Current assignments** (as of vault Phase 2d):

| Vendor | `roles:` |
|---|---|
| Azure-Firewall, AWS-Network-Firewall, GCP-Cloud-Firewall, Check-Point, Cisco-ASA-FTD, Sophos-XG, iptables-nftables | `[firewall]` |
| PAN-OS, FortiGate | `[firewall, sd-wan]` |
| Juniper-SRX, OPNsense, pfSense, VyOS | `[firewall, router]` |
| Zscaler | `[sse]` |

## Conventions

- **YAML frontmatter** is mandatory on every page. See the contract above.
- **Dates** are ISO 8601 (`YYYY-MM-DD`).
- **Wikilinks** for everything internal: `[[Services/Azure/Azure-Firewall]]` or `[[Services/Azure/Azure-Firewall|Azure Firewall]]` for a display label.
- **External URLs** for vendor docs, blog posts, RFCs, lab repos.
- **Source pointers, never duplication.** Don't restate Microsoft Learn / AWS docs / GCP docs content; summarize and link.
- **Confidence**: mark uncertain claims with `(?)` or `[needs confirmation]`.
- **Filenames**: TitleCase with hyphens (`Azure-Firewall.md`, not `azure_firewall.md` or `azure-firewall.md`). One exception: `_migrated/<specialist>/<skill>.md` keeps the kebab-case skill name verbatim for traceability.
- **No customer data, no PII, no internal-only content.** This vault ships in a public npm package.
- **Scope discipline**: cloud networking only. Compute, storage, identity, AI belong elsewhere.
- **Every analysis-bearing page ends with**: `> Analysis only — verify against vendor documentation before applying.` (matches the runtime guardrail.)

## Retrieval contract (how `cn_search` reads the vault)

The Phase-3 `cn_search` tool builds a **MiniSearch (BM25) index** at session start over `vault/**/*.md` (excluding `_migrated/`). Field weighting:

| Field | Weight | Notes |
|---|---|---|
| `name` (frontmatter) | 5 | exact display name |
| `tags` (frontmatter) | 3 | array of tags |
| `# <H1>` and `## <H2>` headings | 3 | section titles |
| body paragraphs | 1 | full markdown body |
| outbound `[[wikilinks]]` | 2 | link targets — surfaces topical neighbors |

The `specialists` frontmatter field is used as a **filter**, not a search field — `cn_search({ specialist: "cn_fw", query })` restricts BM25 to pages whose `specialists:` array contains `cn_fw`.

When called with `expand_links: true`, the tool takes the top-K results, parses outbound `[[wikilinks]]` from each result body, fetches the **first H1 + Summary section** of each linked page, and appends them as cheap context. This is the cross-area "join" mechanism — no graph DB needed.

## Authoring workflow

### Add a new Service / Topic / Pattern / Vendor

1. Pick the right directory (`Services/<cloud>/`, `Topics/`, `Patterns/`, `Vendors/`).
2. Create the file with the appropriate template above.
3. Fill in frontmatter — especially `specialists:` (which personas should retrieve this page).
4. Add `[[wikilinks]]` to related pages — bidirectional links should be reciprocated when relevant.
5. Run the link checker (Phase 3) to catch dead links.

### Refactor a `_migrated/` page

1. Read the page; decide its real `type` (service / topic / pattern / vendor) and target directory.
2. Move + rename the file (use a script when bulk-moving — see `tools/`).
3. Rewrite frontmatter against the type-specific template; drop `migration_source` / `original_*` fields; add `specialists:` and `tags:`.
4. Convert in-body references to other migrated content into `[[wikilinks]]`.
5. Dedupe: if multiple `_migrated/` pages overlap with the target (e.g., 8 specialists each have a `troubleshoot` skill), merge the unique content and consolidate into one page or split per-service.
6. Delete the original `_migrated/` file.

When `_migrated/` is empty, delete the directory and the migration is complete.

## What lives where — quick decision guide

| If the content is about… | It goes in… |
|---|---|
| One cloud service end-to-end (Azure Firewall, AWS NLB, GCP Cloud DNS) | `Services/<cloud>/<Service>.md` |
| A concept that spans multiple services or clouds (BGP, hybrid DNS, IP planning, TLS offload, egress cost, IPv6 transition) | `Topics/<Topic>.md` |
| A reusable architecture (hub-and-spoke, vWAN secured hub, three-tier web, multi-region active-active) | `Patterns/<Pattern>.md` |
| A specific vendor's product/syntax (PAN-OS, FortiGate, Check Point) | `Vendors/<Vendor>.md` |
| A specific dated experiment with a repro and outcome | `Labs/<YYYY-MM-LabName>.md` |
| Specialist persona / workflow / guardrails (the 20 "who am I" definitions) | **Not in the vault.** Stays in `extensions/network-desk/specialists/<name>/agents/<name>.md`. |
